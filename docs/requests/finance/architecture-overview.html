The Finance Hub

How the specs we've written translate into a real architecture — what the finance module will be able to do, and how it will work.

Synthesized from docs/requests/finance/** + notes/finance-corpus/00-inbox/** · branch finance-specs · 2026-06-20

TL;DR

The finance hub is five specs that all serve one moment: the recurring decision to "put $X into these names this week." Around that moment sit four lenses — ingestion (how much can I deploy?), research (into what, and why?), market data (at what price / how has it moved?), and strategy & planning (the actual deployment plan).

The whole system runs on one rule: tools own facts and arithmetic; the agent owns judgment and prose. There is no UI — Claude is the frontend, and every capability is a plain Python function exposed over CLI and MCP. Every number traces to imported data or a deterministic tool result, never model freehand.

Status today: this is a designed system. src/finance_hub/ now contains the installable package skeleton. Ingestion (Slice 1) and market-data acquisition are implementation-ready; research and strategy are sharpened scaffolds; analytics is design-ahead.

0The 60-second version

You asked three things: what files do we have and how are they structured, how does the spec content become an architecture, and what will the finance module actually do and how. This document answers them in that order. Start here for the shape; drill into any section for the substance.

THE CORE

The emotional core sets everything. The whole hub exists to make one recurring decision defensible and low-effort: the deployment moment that fuses budget + thesis + market — "deploy $X into these sleeves this cadence." Budgeting exists so that decision rests on real surplus; research exists so it rests on a cited thesis; market data exists so it rests on a real price. Everything else ranks below the planner.

Four lenses feed one planner. Each is independently useful and independently buildable, and each hides its risky dependencies behind a narrow seam so the others never see a vendor API or an unbuilt layer:

① Ingestion

Statements → reconciled budget facts → a confidence-gated historical surplus. Answers "how much could I deploy?"

② Research

Top-down industry → key players → cited thesis. Produces approved candidates + evidence. Answers "into what, and why?"

③ Market data

Provider bars → grounded prices + (later) metrics. Answers "at what price, and how has it moved?"

④ Strategy & planning

Promote research → versioned strategy → deterministic deployment plan. Answers "so what do I buy this week?"

1How the hub works (the mechanism under every spec)

Before the finance specifics, the one architectural idea they all inherit. finance-hub keeps a headless tool registry. There is no user interface; the agent (Claude) is the frontend. A "capability" is a plain @tool-decorated Python function that the registry exposes over both a CLI and an MCP server. This single reframe resizes the entire build.

The division of labour

Everything in these specs is sorted by one cut — the thing an LLM can't do reliably goes in code; the thing we'd be tempted to code but shouldn't stays with the agent:

Tools own (code)The agent owns (runtime)
Persisted facts, numbers, stateJudgment, "guidance," strategy authoring
Deterministic arithmetic & reconciliationReading a PDF statement / a 10-K
Real data fetches (prices, filings)Mapping an industry to its key players
Validation, idempotency, provenanceForming and writing the thesis prose

The consequences are where the build shrinks: statement ingestion needs no PDF parser (the agent reads it and calls record_transactions); "cost-cutting guidance" is not a tool (it's the agent reasoning over spending_summary()); the "DCA strategy builder" is not a UI (it's a persisted strategy object the agent helps author).

NORTH STAR

Every number traces to imported data or a deterministic tool result — never model freehand. Research extends it from numbers to claims: every thesis claim cites a source (a filing, a trusted analysis, a price, a metric). This is what makes a deployment decision defensible on both the money side and the thesis side.

The implementation shape every spec follows

  • Namespace + home: user-facing tools live under finance.*; finance owns its fin_* tables under src/finance_hub/, using finance-owned store helpers.
  • Pure core + thin wrapper: pure transformations (money parsing, normalization, reconciliation, metric math) live outside the registered functions and are unit-tested directly. The @tool wrapper stays thin: load state → call pure fn → persist → return a structured response.
  • Storage split: SQLite owns the numbers (transactions, prices, weights), markdown / the vault owns the prose (theses, research notes), and generated readouts combine both.
  • Finance-owned migrations: a fin_schema_migrations table — not the global PRAGMA user_version. FK + CHECK + indexes, with PRAGMA foreign_keys = ON per connection.

So "the finance module" is, concretely: a set of fin_* SQLite tables, a set of pure Python helpers, and a set of thin finance.* tool wrappers — all reachable by the agent through CLI and MCP. No server, no UI, no framework.

3How the lenses connect

The four lenses are not a pipeline you run start-to-finish — they are three independent evidence producers that converge on the planner. Each handoff is a narrow, versionable artifact, never a shared mutable blob:

Ingestion
statements → reconciled facts → historical-surplus evidence → (later) capital composition
Research
industry → key players → cited thesis → approved candidates → explicit promotion
Market data
provider bars → fin_price_bars → price envelopes + (later) metrics
▼   all three converge   ▼
Planner
plan_deployment(deployable_capital, cadence) reads strategy + capital + holdings + prices → a reviewable deployment plan

The signature the whole system builds toward makes the convergence concrete:

plan_deployment(deployable_capital, cadence)
  reads:  deployable_capital   ← later composition over BUDGET evidence
          approved strategy     ← explicit promotion from RESEARCH candidates
          holdings              ← later holdings slice
          prices                ← market data (finance.prices(...) PriceEnvelope)
KEY SEAM

Research state is never planner state. Discovering a company and tagging it to a theme does not make it deployable. Only a user-confirmed finance.promote_to_strategy(...) snapshots selected themes, weights, and approved instruments into a new immutable strategy version. This is the firewall between "exploring" and "committing capital."

4What the finance module will be able to do

Translated from the specs into plain capabilities, lens by lens. (Tool names are the actual finance.* functions the specs define. Expand each for the full surface.)

① Ingestion — turn statements into defensible budget evidence

Import Chase checking/credit CSVs deterministically; record agent-extracted rows from PDFs or unsupported formats through the same canonical engine; reconcile transactions against asserted statement balances; report spending and a confidence-gated historical surplus. Re-importing the same file is safe (no double-counting), nothing is ever silently dropped, and corrections supersede facts while preserving the raw evidence.

Ingestion tool surface
ToolDoes
import_csvDeterministic Chase CSV import through the canonical engine
record_transactionsPersist agent-extracted rows (PDF / unsupported formats)
balance_checkReconcile statement intervals; log gaps (never auto-repairs)
match_transfers / record_transferLink internal two-leg transfers
spending_summaryIncome, spend, adjustments, transfers, by category
historical_surplusConfidence-gated average monthly surplus over the reconciled window
add_account · add_category_rule · supersede_transaction · list_issues …Configuration, corrections, issue review

The honesty gate: historical_surplus only populates planner_capital_cents when confidence is high (a common, fully-reconciled window across all included accounts); otherwise it returns NULL. It explicitly does not yet claim the amount is safe to invest.

② Research — explore top-down and preserve a cited thesis

Walk industry → sub-segment → key players → deep research, persisting the durable residue at each step: the theme map, the watchlist (with role + optional conviction), the thesis note (in the vault), the important dates, and the cited sources. Then compose a grounded, cited theme brief / instrument brief and export it as canonical markdown + derived static HTML.

Research tool surface & the agent/tool split

The agent does the research; the tool persists the residue:

StepAgent (judgment)Tool (durable fact)
Industry sweepmaps the landscape (web)set_theme
Key playersnames companies, with cited reasoningmap_instruments → watchlist

What matters nextfinds the datesrecord_event

Commitdecides to fundpromote_to_strategy → planner

There is no discovery algorithm, scraper, or NLP — exactly as the budget lens has no PDF parser. The lens is "a persistence + retrieval + provenance layer over agent-driven research."

③ Market data — ground every price, then (later) every metric

Fetch daily OHLCV bars from a provider (free yfinance first), normalize and store them in fin_price_bars, and hand consumers a small PriceEnvelope via finance.prices(...) — never a raw vendor payload. A manual finance.snapshot() loop refreshes the active universe so you can "see what shifts." Later, the analytics half computes returns, volatility, drawdown, momentum, and — the headline — allocation drift.

④ Strategy & planning — commit, then plan deterministically

Snapshot approved research into a versioned strategy (sleeves + target weights + eligible instruments), and compute a deterministic deployment plan over strategy + capital + holdings + prices. Strategy authoring never executes trades; the planner never invents a number.

ABSOLUTE

Across every lens, inherited and non-negotiable: no trade execution, money movement, or order placement, ever. The system plans and explains; a human acts.

5How the specs translate into data architecture

The specs are careful about where each kind of truth lives. Three modelling decisions carry most of the weight.

5.1 The shared instrument contract

fin_instruments is a deliberately small shared reference — ticker-centric, US-listed, USD stock | etf. Research annotates it (discovery), market data prices it, strategy snapshots approved selections of it. Each layer adds its own tables rather than widening this one into a catch-all. That's how the layers stay independently evolvable.

LensOwns these fin_* tables
Ingestionfin_accounts, fin_transactions, fin_balances, fin_ingest_runs, fin_source_records, fin_ingest_issues, fin_category_rules, fin_transfer_groups
Researchfin_themes, fin_theme_instruments, fin_research_sources, fin_research_source_links, fin_events
Market datafin_price_bars (active), fin_fetch_log, fin_metrics (deferred)
Strategyfin_strategies, fin_strategy_versions, fin_strategy_sleeves, fin_strategy_instruments, fin_deployments
Sharedfin_instruments — the narrow join across research / market data / strategy

5.2 Numbers vs prose vs generated readouts

A three-way split applied everywhere:

Markdown owns research thinking.       (the vault — theses, evolving reasoning)
SQLite owns structured facts & relationships.   (fin_* tables — filterable, validated)
Generated readouts combine both.       (replaceable; marked "generated — do not edit")

The deciding rule when the model grows: if it needs filtering, validation, or deterministic composition → SQLite; if it needs interpretation, nuance, or revision → Markdown. Prose is never a TEXT blob — SQLite stores a relative note_path and the agent authors the file.

5.3 The graded-provenance envelope

This is the literal mechanism of the north star. A quantitative value cannot cross a seam as a bare number — it travels in an envelope that names its source and its trust grade:

{ "value": 28.5, "source": "fmp", "grade": "screening", "as_of": "2026-05-30" }
GradeMeansFrom
decisionprimary-source ground truth — safe for a thesis-critical claimedgartools / SEC EDGAR filings
screeningdirectional only — never decision-gradeFMP aggregator, agent web reads, and all price-derived values

Two rules fall out: a composite's grade = the lowest grade of its inputs (a score built on FMP numbers is screening), and you compose, don't blend — different columns of a table may come from different sources, but you never average two vendors into one "consensus," and never build one price series from mixed vendors. This is what makes the multi-source comparison table (the "pie") honest: each cell self-identifies.

PRECISION

Money is signed integer cents; prices are integer micro-dollars (PRICE_SCALE = 1_000_000). Both parse through Decimal at the boundary — never float, never REAL. close (raw) is the planner/valuation anchor; adj_close (vendor-adjusted, refreshable) is the return-math field.

6The market-data stack (the clearest example of the layered design)

Market data is specced as one stack of seven layers. Each layer reads only from the layers below it, writes its own tables, and is reachable through a stable seam — which is exactly what turns a new metric or a vendor swap into an additive change rather than a rewrite. It's the architectural pattern of the whole hub, made explicit.

L6Simulation / backtest"what if I'd DCA'd this strategy for 2 years?"far future
L5Event-response analyticsearnings reaction, [-5,+5] return, post-event driftdeferred
L4Cross-sectional analyticsrank / screen / compare players within a theme — finance.compare()deferred
L3Aggregated metricssleeve & portfolio rollups — allocation drift, concentrationdeferred
L2Derived metricsreturns, vol, drawdown, momentum (per instrument)deferred
▲ analytics reads · acquisition writes ▲   seam: fin_price_bars
L1Stored observationsfin_price_bars — daily OHLCV + refreshed adj_closeactive
L0ProvidersPriceProvider.fetch_daily_bars → yfinance | FMP | SEC EDGARactive

The split into two sub-specs mirrors a real boundary: acquisition (L0–L1) is ops-flavoured (adapters, secrets, retries, rate limits) and runs light; analytics (L2–L6) is compute-flavoured (metric math, notebooks) and pulls the heavy pandas/numpy/jupyter extra. fin_price_bars is the one contract between them — reads go strictly downward.

Where the build line sits

The market-data team drew an explicit "first push" line after weighing four candidate stopping points. The reframe mattered as much as the choice:

CandidateAddsVerdict
Aon-demand daily-bar fetch + finance.prices(...) envelopes + cache + overridesSlice 1 — review checkpoint
BA + finance.snapshot() manual universe loop → "see what shifts" beginsSlice 2 — the drawn line
Automatescheduling (launchd, DST, market calendar, missed-run backfill)parallel later gate
Analyzethe L2 metrics step (owned by analytics)parallel later gate

The insight: A→B is a near-term spine, not a fork (A alone only writes a bar when you happen to fetch — its series is incidental; B makes it real and is cheap on top). After B, Automate and Analyze become parallel, independently-triggered gates — not points 3 and 4 on one line. And the headline "what shifted" metric (allocation drift) is gated outside this subsystem entirely, because it needs holdings + strategy weights that arrive in later slices.

7The load-bearing boundary decisions

Six decisions from the spec map hold the whole architecture together. They're the answers to "why isn't this simpler?" — each prevents a specific way the system could rot.

#DecisionWhat it prevents
1Research and strategy are separate. Discovery changes research state only.A web search silently changing what you're eligible to buy.
2Promotion is explicit. A user-confirmed action snapshots candidates into a versioned strategy.Conviction quietly becoming allocation.
3Ingestion doesn't over-claim capital. Slice 1 produces historical surplus evidence, not deployable capital.Confusing "I had surplus last quarter" with "this is safe to invest."
4Market data is a shared subsystem, not a workflow. Research/planning read narrow seams.Vendor APIs leaking into every lens.
5The agent owns judgment; tools own facts & arithmetic.Prose or "agent confidence" turning into an allocation.
6Research readouts have one source of truth. Markdown is canonical; HTML is generated from the same model.Two divergent copies of a thesis; hand-edited "facts."
THE POLICY

Modular evolution, not a platform up front. Each layer stays independently evolvable: narrow versionable handoffs, additive migrations, provider logic behind adapters, immutable snapshots where history matters. Add fields and tables when a demonstrated workflow needs them — avoid speculative abstractions and vendor-shaped core models.

PACING

A recurring corpus rule shows up in every spec: build the narrowest real slice, then stop for review. Pure helpers are TDD'd outside the @tool wrappers; wrappers stay thin. The first commit of each lens is a single end-to-end path, not the whole surface.

9Glossary — the terms that recur

TermMeaning
ThemeA research/exploration object (e.g. compute, silicon). You explore many you'll never fund.
SleeveA theme you've committed capital to — theme + target_weight inside a specific strategy version. The join point of thesis, capital, and cadence.
Historical surplusReconciled evidence of average monthly surplus over a confidence-gated window. Evidence, not yet deployable capital.
Deployable capitalA later composition: surplus + liquid cash + reserves − obligations ± explicit override. Not built in Slice 1.
