2. What Is Real Today

There are two parallel structures in the repo: an implemented finance tool runtime, and a largely unimplemented finance domain that has been designed to plug into that runtime.

Shared runtime exists

  • finance_hub.runtime.registry registers tools once.
  • finance_hub.cli exposes them as finance run ....
  • finance_hub.mcp_server exposes the same tool set to agents.
  • The runtime is installed as the finance and finance-mcp commands.
implemented

Finance architecture exists

  • Specs define tables, seams, build order, and tool contracts.
  • North-star rules around provenance are repeated consistently.
  • Cross-spec boundaries are surprisingly clean.
well specced

Finance code does not yet exist

  • src/finance_hub/ now contains package placeholders.
  • No real finance ingestion, market-data, research, or strategy tools are implemented yet.
  • The finance CLI/MCP commands are wired to the placeholder registry.
runtime gap
Important distinction: if you read only the docs, it feels like a package already exists. If you read only the code, finance is still mostly scaffolding. The correct understanding is: the repo has a designed finance subsystem waiting to be implemented on top of an already-existing CLI/MCP tool runtime.

3. How the Specs Translate to Architecture

The finance design is not one monolith. It is a set of narrow subsystems that feed a future planner. The architecture is intentionally layered so each slice can ship without forcing the others to exist.

Core design principles that repeat across specs

  • The agent owns judgment and prose; tools own durable facts and deterministic arithmetic.
  • Every number or claim needs provenance.
  • Cross-layer state changes must be explicit.
  • Provider-specific logic stays behind adapters and seams.
  • Defer complexity until a named workflow demands it.

Why the architecture feels coherent

  • Research does not mutate strategy implicitly.
  • Market data is a subsystem, not a user-facing workflow.
  • Planner inputs are deliberately narrow and versionable.
  • The specs consistently separate build-now slices from design-ahead slices.

4. What the Finance Package Will Probably Look Like

The docs imply a package shape even though the code is not there yet. The likely structure is a finance-specific implementation layer under src/finance_hub that mirrors the durable spec split and registers tool wrappers in one place.

Likely code organization

src/finance_hub/ __init__.py # tool registration surface store.py # finance db access / connection helpers migrations.py # fin_schema_migrations runner ingestion/ importers.py reconcile.py categories.py balances.py tools.py research/ themes.py notes.py sources.py events.py briefs.py tools.py market_data/ providers.py acquisition.py snapshot.py analytics.py # later tools.py strategy/ versions.py planner.py holdings.py tools.py

Design pattern from the current repo

  • Pure functions should hold domain logic.
  • Thin @tool wrappers register the public capability surface.
  • The same tools then become callable from shell and MCP without extra glue.
  • Finance-specific persistence stays finance-owned under finance_hub.store.

In other words, finance is not being designed as a web app. It is an installable agent tool package with a CLI and MCP runtime.

Key structural insight: the durable specs are already acting like the package's internal module boundaries. If implementation ignores that and collapses everything into one a single catch-all module, it will fight the design almost immediately.

6. How It Will Work End to End

The intended runtime model is straightforward: one tool registry, one database, several finance-owned domains, and a set of flows that hand off explicit artifacts to one another.

What the planner eventually consumes

  • Budget evidence, not raw statements
  • Approved strategy, not raw research watchlists
  • Grounded price envelopes, not vendor API responses
  • Later holdings snapshots and strategy weights

What this buys you

  • You can ship ingestion before planning.
  • You can ship qualitative research before quant analytics.
  • You can swap providers without rewriting planner logic.
  • You can expose all of it to agents through one stable namespace.

7. Practical Reading of the Build Order

If you want to understand what should happen next in the codebase, the specs point to a fairly clear implementation order.

Order Work Why it is next
1 Build finance-owned persistence and migration scaffolding Every spec assumes fin_* tables and finance-owned migration control.
2 Implement ingestion slice 1 tools This is the most complete implementation contract and creates the budget evidence side.
3 Implement market-data acquisition Slice A finance.prices(...) is the first hard dependency for the planner and for research briefs with pricing context.
4 Implement the qualitative research layer The research spec is already useful before metrics and fundamentals activate.
5 Sharpen and implement strategy state + promotion seam The planner cannot be honest until research-to-strategy handoff is explicit.
6 Add market-data snapshot loop and later analytics gates Only once there is a bar history worth analyzing.

Most important unresolved gap

Strategy is the least-finished durable spec. The package already knows how to produce budget evidence, research evidence, and price inputs, but the actual planner-owned state model still needs extraction from requirements-dump.md into a stronger contract.

Most likely first meaningful demo

A realistic early end-to-end demo would be: import Chase CSVs, reconcile balances, compute historical_surplus, record a theme and a few instruments, fetch current price envelopes with finance.prices(...), and render a grounded theme brief.
