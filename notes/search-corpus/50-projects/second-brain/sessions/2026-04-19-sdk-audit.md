# SDK Audit: TypeScript RAG Framework for Second Brain

> Which framework best fits a CLI-first Obsidian vault RAG system with future agentic workflows?
>
> **Decision context:** We need markdown ingestion, embedding, vector search, conversational Q&A, streaming, multi-model support (Claude + Ollama + OpenAI), and eventually agentic claim extraction.

---

## Top 3 Recommendations

### 1. LlamaIndex.TS — Best RAG-first fit

LlamaIndex.TS is purpose-built for RAG. Unlike LangChain (which is a general-purpose LLM toolkit that *also* does RAG), LlamaIndex was designed from day one around document ingestion, indexing, and retrieval. The TypeScript version is not a port — it was rewritten natively for JS/TS and runs on Node, Bun, Deno, and Cloudflare Workers.

**Why it wins for us:**
- **Markdown ingestion is first-class.** Built-in document loaders, multiple chunking strategies (sentence, paragraph, semantic), and index types (vector, summary, keyword). No need to hand-wire a document pipeline.
- **Retrieval accuracy.** Benchmarks show ~92% retrieval accuracy vs LangChain's ~85% on standard evaluation sets. For a second brain where retrieval quality *is* the product, this matters.
- **Vector store breadth.** ChromaDB, pgvector, Pinecone, Weaviate, Milvus, Qdrant — start local, graduate to hosted with a one-line swap.
- **Agentic path.** The Workflows API supports agentic patterns when we get to claim extraction. Not as rich as LangGraph, but sufficient.
- **LlamaParse** (optional) handles complex document parsing if we ever ingest PDFs or slides.

**Trade-offs:**
- Smaller community than LangChain (though growing fast)
- Fewer blog posts / StackOverflow answers for edge cases
- Less flexibility for non-RAG LLM tasks (LangChain is broader)

### 2. Mastra — Best for agent-heavy future

Mastra is a TypeScript-native AI framework (1.0 shipped January 2026, 22k+ GitHub stars, 300k+ weekly npm downloads) built by the ex-Gatsby team. It combines RAG, agents, and workflow orchestration in a single cohesive framework.

**Why it's compelling:**
- **RAG + Agents + Workflows unified.** No stitching LangChain + LangGraph + separate tools. One framework handles document chunking (recursive, sliding window), embedding (local via `@mastra/fastembed` or hosted), retrieval, and agentic workflows.
- **Graph RAG and ReAG** (reason-over-documents) are built in — relevant for our claim relationship classification.
- **Mastra Studio** (localhost:4111) gives visual debugging of agent runs — useful for iterating on claim extraction prompts.
- **Built-in evals** (toxicity, bias, relevance) — aligns with our eval harness requirement.
- **Wraps Vercel AI SDK** internally, so all AI SDK providers work automatically.

**Trade-offs:**
- Youngest of the three — 1.0 is only 3 months old
- Fewer vector store integrations than LlamaIndex (pgvector, Pinecone, Qdrant, MongoDB)
- Less battle-tested in production RAG workloads
- Moving fast = potential breaking changes

### 3. LangChain.js + LangGraph — Most ecosystem, most overhead

LangChain.js 1.0 (November 2025) stabilized the API after the v0.x era of frequent breaking changes. Combined with LangGraph for agentic workflows, it's the most comprehensive toolkit available.

**Why it's still in the running:**
- **Largest ecosystem.** Most integrations, most examples, most community answers.
- **Markdown loaders + text splitters** work well out of the box.
- **LangGraph** is the best graph-based agent orchestrator — stateful, supports cycles, checkpointing, human-in-the-loop. Ideal for iterative claim extraction.
- **LangSmith** adds production observability (traces, evals, monitoring).
- **Vector store breadth** comparable to LlamaIndex.

**Trade-offs:**
- **Heaviest abstraction layer.** The Runnable/LCEL/Chain/Agent abstraction stack has a real learning curve.
- JS version historically lags behind Python in features and docs
- For a straightforward RAG pipeline, most of LangChain's machinery is unused weight
- LangGraph adds complexity you don't need until the agentic phase

---

## Verdict

**Start with LlamaIndex.TS.** It is the most focused tool for what we're building *right now* — RAG over markdown files. It has the best retrieval primitives, the cleanest TypeScript DX for document pipelines, and enough agentic support for later phases.

**Watch Mastra closely.** If the agentic claim extraction phase becomes the center of gravity, Mastra's unified agent + RAG + workflow model may justify a migration. Revisit after Phase A (search) ships.

**Use Vercel AI SDK as the streaming/UI layer** regardless of which RAG framework we pick. LlamaIndex.TS and Mastra both interoperate with it. Add it when we build a web frontend.

**Add Instructor for structured extraction** when we build claim extraction — it's a lightweight Zod-based layer that works with any provider.

## Recommended Stack

```
LlamaIndex.TS          → RAG pipeline (ingestion, chunking, indexing, retrieval)
Vercel AI SDK          → Streaming layer + multi-provider model calls (add with web UI)
Instructor             → Structured claim extraction (add in Phase B)
Anthropic prompt cache  → Cost optimization for large retrieved contexts
ChromaDB (local)       → Initial vector store (migrate to pgvector later)
```

---

## Decision (2026-04-19)

**Confirmed: Vercel AI SDK + hand-rolled agent loop + LlamaIndex.TS (utility only for Phase B ingestion).** This is a two-way door — the data layer is portable across all frameworks. Updated in [[03-tech-stack#3. Agent Framework]].

Key factors:
- The project is still in domain discovery — heavy frameworks hide decisions we need to make consciously
- LlamaIndex.TS earns its keep for markdown loading/chunking, not as a framework
- Mastra is the leading candidate if we need workflow orchestration later (Vercel AI SDK foundation = portable LLM layer)
- LangChain has the deepest lock-in and heaviest overhead — ruled out
